## 1. Objetivo do L3.0

Provar o pipeline **t28 → score canônico → LLM → DB Análises PHI** end-to-end no
nível **campanha** (o que tem dado para TODOS os clientes, inclusive PMAX onde
`t28_adset=0`, e é a unidade primária de relatório). Uma análise de campanha
real (CLI-4) materializada como page no Notion. Depois fan-out (L3.1 ad/adset,
L3.2 cliente, L3.5 entrega completa).

**Fora do L3.0:** ad/adset/cliente levels, update de properties dos DBs
existentes, Telegram, loop Demandas (ADR-22), sink BQ de análises. Tudo isso é
L3.1+/L3.5.

## 2. Resolução do score (refina a decisão #3)

"Determinístico" **não é recalcular** o PHI·Mídia no L3 — seria violar a
autoridade única do **ADR-003** (o `PHI-Pipeline_v2` é o dono do score). O
PHI·Mídia canônico já existe em **`phi_prod.phi_score_history`** (calculado 07:00
pelo Pipeline_v2 via fórmula ADR-004 v2: MIV/MAS/TSS/FIS/ES/RS).

**Decisão de design:** o L3.0 **lê** o PHI·Mídia de `phi_score_history`
(determinístico, reproduzível, auditável) e o passa ao LLM como insumo. O LLM
produz só o **qualitativo** (insight + recomendações + flags + severidade) — não
atribui score.

> ⚠️ A verificar na build: schema/chaves de `phi_score_history` (presumível join
> com `t28_campaign` por `client_id` + `campaign_id` + data). Se a granularidade
> não casar (ex.: Pipeline_v2 keya diferente), ajustar o join. Não é fork de
> design — é detalhe de implementação.

## 3. Arquitetura L3.0

```
WF-T28-Orquestrador-Analises (NOVO, schedule próprio)
  Schedule Trigger (semanal D-7 / mensal D-30, espelha o Agregador)
  → lê clientes da janela (DB Clientes Notion ou lista)
  → para cada (cliente, campanha) com t28_campaign na janela:
       lê t28_campaign (métricas + contexto)
       lê PHI·Mídia de phi_score_history (canônico)
       → Execute Workflow Trigger →  WF-T28-Analise-Campaign (sub-WF)
                                        LLM Gemini Pro (tiering): insight/recs/flags/sev
                                        → escreve 1 page em DB Análises PHI
```

- **2 workflows** (ADR-23: análise separada do ETL):
  - **`WF-T28-Orquestrador-Analises`** — orquestra: lê t28 + score, fan-out por
    campanha via Execute Workflow Trigger, controla janela/quota.
  - **`WF-T28-Analise-Campaign`** (sub-WF reutilizável) — recebe 1 campanha +
    score, chama o LLM, escreve a análise. Reutilizável por nível depois
    (mesma estrutura para adset/ad no L3.1, espelho do ADR-24).
- **Gatilho próprio** (decisão #2): não encadeado ao Agregador. Roda após o
  Agregador popular `t28_*` (cadência própria). Desacoplado/retryável (ADR-23).
- **Tiering** (ADR Tiering): campaign-level = **Gemini Pro** (análise densa).
  Ad/adset (L3.1) = Flash. **Dependência: credencial Gemini no n8n** (pendência
  conhecida do `WF-EXEC-Orquestrador`; precisa estar configurada).

## 4. Framework de análise campaign-level ⚠️ (REVISAR — IP de domínio)

> Esta seção é o coração do L3 e precisa do seu olhar. É um strawman.

### 4.1 Input do LLM (montado pelo Orquestrador, determinístico)
- **Identidade:** cliente, campanha (nome/id), janela, business_date.
- **Score canônico:** PHI·Mídia (0–100) + leitura (CRITICAL/WARNING/GOOD/EXCELLENT)
  de `phi_score_history`. Se disponível, os 6 componentes (MIV/MAS/TSS/FIS/ES/RS).
- **Métricas t28_campaign:** impressions, clicks, cost, conversions, conv_value,
  cpm, cpc, ctr, cvr, cpa, cpl, roas, impression_share, budget_lost_is.
- **Contexto de negócio:** objetivo, modelo_negocio, metrica_mae (CPA|ROAS),
  meta_metrica_mae (alvo), margem_contribuicao_pct, ticket_ltv.
- **Qualidade:** volume_suficiente, source_status (quais fontes degradaram).
- **Tendência (se já houver histórico t28):** D-7 vs D-7 anterior (delta dos KPIs).

### 4.2 Output do LLM (estruturado — Structured Output Parser)
```
{
  "insight": "diagnóstico curto do estado da campanha (2-4 frases)",
  "recomendacoes": [ {acao, racional, esforco: baixo|medio|alto, impacto_esperado} ],
  "flags": ["cpa_acima_meta", "impression_share_baixo", "budget_lost", ...],
  "severidade": "info|atencao|critico",
  "evidencias": ["métricas que sustentam o diagnóstico"]
}
```
- **O LLM NÃO emite score.** Severidade é qualitativa (≠ score).
- **Guardas (IA Cognitiva, ADR-21):** se `volume_suficiente=false` → análise diz
  "VOLUME INSUFICIENTE" e não dispara recomendação agressiva (Tema 10). Distinguir
  ruído de leilão/sazonalidade de problema real (Tema 19). Não otimizar 1 KPID só
  (Tema 05 — usa o vetor de KPIs + metrica_mae).

### 4.3 Taxonomia de flags + severidade ⚠️ (definir com Olavo)
Strawman inicial (determinístico, computado no Orquestrador OU sugerido pelo LLM):

| Flag | Gatilho (exemplo) | Severidade base |
|---|---|---|
| `cpa_acima_meta` | cpa > meta_metrica_mae × 1.2 (se metrica_mae=CPA) | atenção/crítico |
| `roas_abaixo_meta` | roas < meta_metrica_mae (se metrica_mae=ROAS) | atenção/crítico |
| `impression_share_baixo` | impression_share < 0.5 | atenção |
| `budget_lost` | budget_lost_is > 0.2 | atenção |
| `ctr_baixo` | ctr < benchmark do objetivo | info |
| `sem_conversao` | conversions=0 com cost>limiar | crítico |
| `volume_insuficiente` | volume_suficiente=false | info (bloqueia recs) |

**Decisão pendente:** as flags são (a) determinísticas no Orquestrador (regras),
(b) sugeridas pelo LLM, ou (c) híbrido (regras geram flags; LLM prioriza/explica)?
Recomendo **(c)**: regras determinísticas para reprodutibilidade + LLM para
narrativa e priorização.

## 5. Schema DB Análises PHI (Notion, novo) — refina ADR-27

Properties (campaign-level; nível/entidade generalizam no fan-out):
- `Titulo` (title) — ex.: "[campaign] CLI-4 / Campanha X — D-7 (2026-06-28)"
- `Nivel` (select: ad | adset | campaign | cliente) = campaign
- `Cliente` (relation → DB Clientes) — ou rich_text no L3.0 se relation atrasar
- `Entidade` (relation → DB Campanhas) — idem
- `Janela` (select: D-1 | D-3 | D-7 | D-30)
- `Business Date` (date)
- `PHI Midia Score` (number) — o canônico de phi_score_history
- `Leitura` (select: CRITICAL | WARNING | GOOD | EXCELLENT)
- `Flags Ativas` (multi-select)
- `Severidade` (select: info | atencao | critico)
- `execution_id` (rich_text) — EXEC-ORQ-*
- `Versao Contract` (rich_text)
Body: Insight (LLM) · Recomendações (LLM) · Evidências (métricas).
