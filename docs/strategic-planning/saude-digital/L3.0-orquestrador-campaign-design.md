## 1. Objetivo do L3.0

Provar o pipeline **t28 → score canônico → LLM → DB Análises PHI** end-to-end no
nível **campanha** (o que tem dado para TODOS os clientes, inclusive PMAX onde
`t28_adset=0`, e é a unidade primária de relatório). Uma análise de campanha
real materializada como page no Notion. Depois fan-out (L3.1 ad/adset, L3.2
cliente, L3.5 entrega completa).

**Ordem de build (decisão Olavo 2026-06-30):** **plumbing primeiro** (Orquestrador
lê t28 + score, fan-out, resolve relations, escreve page no DB Análises com
framework placeholder/strawman); o **framework de análise §4 é o ÚLTIMO ponto**,

**Fora do L3.0:** ad/adset/cliente levels, update de properties dos DBs

## 2. Resolução do score (refina a decisão #3) — VERIFICADO

"Determinístico" **não é recalcular** o PHI·Mídia no L3 — seria violar a
autoridade única do **ADR-003** (o `PHI-Pipeline_v2` é o dono do score). O
PHI·Mídia canônico já existe em **`phi_prod.phi_score_history`**.

**Schema verificado** (via node `MERGE phi_score_history` do Pipeline_v2 +
handoff A.7b DDL/VIEW):
- **Chave de identidade:** `client_id` + `campaign_id` + `calculated_date` (DATE).
  **Sem `janela`** — é **1 score por (cliente, campanha, data)**.
- **Score PHI·Mídia** = coluna `phi_value` (FLOAT64).
- **Leitura** = coluna `phi_classification` (STRING; enum CRITICAL/WARNING/GOOD/EXCELLENT).
- **Componentes** (ADR-004 v2): `mas, tss, fis, es, rs, os, miv, miv_normalizado,
  priority_score, threshold_used`, + `business_model`, `mas_janela`.
- Filtrar `calculation_status = 'SUCCESS'`.
- VIEW **`phi_score_current`** = último SUCCESS por `client_id+campaign_id`
  (`MAX(calculated_date)` + INNER JOIN), já determinística hoje.

**Decisão de design (leitura do score no L3.0):** ler de **`phi_score_current`**,
join por **`client_id` + `campaign_id`** (sem alinhar data/janela — pega o score
mais fresco, que é o estado de saúde atual da campanha). O `t28_campaign` tem N
linhas por (cliente, campanha, data) — uma por janela — e o mesmo `phi_value`
canônico se aplica a todas. `calculated_date` fica gravado na análise para
rastreabilidade. Se no futuro precisar de score por janela específica, aí sim
ler de `phi_score_history` com filtro de data — mas é detalhe de implementação,
não fork de design.

## 3. Arquitetura L3.0

```
WF-T28-Orquestrador-Analises (NOVO, schedule próprio)
  Schedule Trigger (semanal D-7 / mensal D-30, espelha o Agregador)
  → lê clientes/campanhas da janela (Clientes Database + t28_campaign)
  → para cada (cliente, campanha) com t28_campaign na janela:
       lê t28_campaign (métricas + contexto)
       lê PHI·Mídia de phi_score_current (canônico) por client_id+campaign_id
       resolve relations Notion (Cliente/Campanha) por client_id/campaign_id
       → Execute Workflow Trigger → WF-T28-Analise-Campaign (sub-WF)
                                       LLM Claude (densa): insight/recs/flags/sev
                                       → escreve 1 page em PHI - Análises
```

- **2 workflows** (ADR-23: análise separada do ETL):
  - **`WF-T28-Orquestrador-Analises`** — orquestra: lê t28 + score, resolve as
    relations, fan-out por campanha via Execute Workflow Trigger, controla
    janela/quota.
  - **`WF-T28-Analise-Campaign`** (sub-WF reutilizável) — recebe 1 campanha +
    score + relations, chama o LLM, escreve a análise. Reutilizável por nível
    (espelho do ADR-24).
- **Gatilho próprio** (decisão #2): roda após o Agregador popular `t28_*`,
  cadência própria. Desacoplado/retryável (ADR-23).
- **Tiering (decisão Olavo 2026-06-30):** campaign-level (análise densa) =
  **Claude**; ad/adset (L3.1, alto volume/baixo custo) = **Gemini**.
  **Dependência: credencial Claude/Anthropic no n8n** (substitui a dep. Gemini
  do L3.0; Gemini fica para o L3.1).

> substituído — a versão vigente é o documento do framework.

### 4.1 Input do LLM (strawman — montado pelo Orquestrador, determinístico)
- **Identidade:** cliente, campanha (nome/id), janela, business_date.
- **Score canônico:** PHI·Mídia (`phi_value`) + leitura (`phi_classification`)
  de `phi_score_current`. Se útil, componentes (mas/tss/fis/es/rs/os).
- **Métricas t28_campaign:** impressions, clicks, cost, conversions, conv_value,
  cpm, cpc, ctr, cvr, cpa, cpl, roas, impression_share, budget_lost_is.
- **Contexto de negócio** (da Campanhas/Clientes Database): objetivo,
  modelo_negocio, metrica_mae (CPA|ROAS), meta_metrica_mae,
  margem_contribuicao_pct (Clientes), ticket_ltv (Clientes).
- **Qualidade:** volume_suficiente, source_status (fontes degradadas).

### 4.2 Output do LLM (strawman — Structured Output Parser)
```
{
  "insight": "diagnóstico curto do estado da campanha (2-4 frases)",
  "recomendacoes": [ {acao, racional, esforco: baixo|medio|alto, impacto_esperado} ],
  "flags": ["cpa_acima_meta", "impression_share_baixo", ...],
  "severidade": "info|atencao|critico",
  "evidencias": ["métricas que sustentam o diagnóstico"]
}
```
- **O LLM NÃO emite score.** Severidade é qualitativa (≠ score).
- **Guardas (IA Cognitiva, ADR-21):** `volume_suficiente=false` → "VOLUME
  INSUFICIENTE", sem rec agressiva (Tema 10). Distinguir ruído de problema real
  (Tema 19). Não otimizar 1 KPI só (Tema 05 — usa vetor + metrica_mae).

### 4.3 Flags + severidade ⚠️ (sub-chat define a taxonomia final)
**Decisão #2 (Olavo 2026-06-30): híbrido** — regras determinísticas geram flags
(reprodutibilidade) + LLM prioriza/explica/narra. Strawman atual (semeado no DB):

| Flag | Gatilho (exemplo) | Severidade base |
|---|---|---|
| `cpa_acima_meta` | cpa > meta_metrica_mae × 1.2 (se metrica_mae=CPA) | atenção/crítico |
| `roas_abaixo_meta` | roas < meta_metrica_mae (se metrica_mae=ROAS) | atenção/crítico |
| `impression_share_baixo` | impression_share < 0.5 | atenção |
| `budget_lost` | budget_lost_is > 0.2 | atenção |
| `ctr_baixo` | ctr < benchmark do objetivo | info |
| `sem_conversao` | conversions=0 com cost>limiar | crítico |
| `volume_insuficiente` | volume_suficiente=false | info (bloqueia recs) |

## 5. DB Análises PHI = `PHI - Análises` (Notion, novo) — refina ADR-27

> (`9d6b65e5-c72b-82e7-856d-81bc34933316`) — mesmo lugar do **`PHI - Demandas`**
> (irmão operacional, ADR-22).
> **Convenção da casa** (descoberta no `PHI - Demandas`): property names
> **snake_case sem acento** — porque é o **Orquestrador (n8n) que escreve** o DB;
> chaves sem acento mapeiam direto do JSON do workflow, zero risco de mojibake.
> **Status:** ✓ CRIADO 2026-06-30 (`PHI - ANÁLISES`, databaseId
> `38fb65e5-c72b-80db-a425-e5939fc35c7a`, data source `38fb65e5-c72b-80ff-9543-000b9a7468af`).
