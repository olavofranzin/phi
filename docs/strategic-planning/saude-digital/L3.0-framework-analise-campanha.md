## 1. Decisão — Reuso vs zero

**Decisão:** **reuso parcial.** O `[SOP v1.0] Time de Agentes de Análise de
Campanhas (Managed Agents)` (`378b65e5-c72b-813a-a8d3-dbe01bc8310d`) descreve
um sistema **diferente e mais pesado**: 6 agentes especialistas + Maestro em
**Claude Managed Agents**, sessões com estado, memory store em disco
(`decisoes-phi/`), rodado via script Python (`run_analise.py`) para análises
**pontuais e profundas**, com humano lendo o stream e validando cada decisão.

O L3.0 precisa de outra coisa: **1 chamada estruturada por campanha**, em lote
(fan-out do Orquestrador), sem sessão, sem stream, saída determinística
(Structured Output) para popular 1 page no Notion. São modos de uso
diferentes — não dá para plugar o Maestro num node de workflow.

**O que reusamos do SOP (doutrina, não arquitetura):**
- A persona (analista de tráfego pago sênior) e o princípio de que o agente
  **orienta, não decide/executa** (idêntico ao princípio central do PHI™).
- O marcador epistêmico **`[CERTEZA]` vs `[HIPÓTESE]`** — todo diagnóstico
  distingue fato calculado de leitura interpretativa. Adaptado abaixo como
  campo `confianca` no output estruturado (mais barato que exigir o LLM
  formatar `[TAG]` inline em texto livre).
- O princípio "métrica isolada não define decisão" (SOP §Objetivo + PHI SOP/
  Glossário §4.1) — ver Guarda #3 (§6 abaixo).
- **NÃO reusamos:** os 6 agentes especialistas, o memory store, o ciclo
  O.D.A.E. completo como orquestração multi-turno. Ficam reservados para uma
  eventual análise sob demanda (deep-dive humano), não para o job automatizado
  do L3.0.

## 2. Decisão — Prompt do LLM (Claude)

### 2.1 System prompt

```
Você é o analista de tráfego pago sênior do PHI™ (Google Ads e Meta Ads).
Sua função é interpretar o estado de UMA campanha, cujo diagnóstico
quantitativo (PHI·Mídia) já foi calculado por outro sistema, e produzir uma
leitura qualitativa acionável para o gestor humano da conta.

Princípios inegociáveis:
1. Você NÃO otimiza a campanha. Você orienta; a decisão e a execução são do
   gestor humano (princípio central do PHI™).
2. Você NÃO recalcula nem ajusta phi_value/phi_classification. São fatos
   dados por um sistema à parte (ADR-003, autoridade única do score) — trate
   como verdade, não questione o número.
3. As flags em "flags_ativas" já foram computadas por regras determinísticas
   e são fixas — você NÃO cria, remove ou reclassifica flags. Sua função é
   priorizá-las e explicá-las na narrativa.
4. A severidade em "severidade_base" também já foi computada
   deterministicamente — reproduza-a em "severidade" (não decida um valor
   diferente). Se você julgar que a leitura correta seria outra, registre
   isso como observação dentro de "insight", mas não altere o campo.
5. Métrica isolada nunca define diagnóstico. Leia o vetor completo: métrica-mãe
   (CPA ou ROAS) + impression_share + budget_lost_is + os 6 componentes do
   score (MAS/TSS/FIS/ES/RS/OS) antes de concluir.
6. Distinga ruído de problema real: TSS baixo (< 40) indica tendência
   recente/instável — trate a leitura como "hipótese", não "certeza". TSS
   alto com desvio mantido é uma leitura mais confiável.
7. Se "volume_suficiente" = false: a única recomendação permitida é continuar
   observando ou ampliar a janela de coleta. Não sugira mudanças de
   orçamento, pausa, corte de público/palavra-chave ou qualquer ação
   agressiva. Comece o "insight" com "VOLUME INSUFICIENTE".
8. Máximo de 3 recomendações, ordenadas por impacto esperado.
9. Escreva em português do Brasil, tom direto e profissional, 2 a 4 frases
   no "insight". Sem jargão desnecessário, sem enrolação.
10. Responda SOMENTE com o schema estruturado fornecido. Nenhum texto fora
    dele.
```

## 3. Decisão — Taxonomia final de flags + gatilhos + severidade base

Híbrido confirmado: **regras determinísticas geram as flags** (reprodutível,
auditável); **o LLM prioriza, explica e narra** — não redefine.

| Flag | Gatilho exato | Severidade base |
|---|---|---|
| `volume_insuficiente` | `qualidade.volume_suficiente === false` | **info** (guarda — força severidade final, ver §4) |
| `score_indisponivel` | `score.phi_value == null` (LEFT JOIN sem match em `phi_score_current`) | info |
| `tendencia_instavel` *(novo)* | `score.tss != null && score.tss < 40` | info (modificador de confiança, não eleva severidade — ver §8) |
| `cpa_acima_meta` | `metrica_mae='CPA' && meta>0 && cpa != null && cpa > meta*1.2 && cpa <= meta*1.5` | atenção |
| `cpa_muito_acima_meta` | `metrica_mae='CPA' && meta>0 && cpa != null && cpa > meta*1.5` | crítico |
| `roas_abaixo_meta` | `metrica_mae='ROAS' && meta>0 && roas != null && roas < meta && roas >= meta*0.5` | atenção |
| `roas_muito_abaixo_meta` | `metrica_mae='ROAS' && meta>0 && roas != null && roas < meta*0.5` | crítico |
| `impression_share_baixo` | `impression_share != null && impression_share < 0.5 && impression_share >= 0.3` | atenção |
| `impression_share_muito_baixo` | `impression_share != null && impression_share < 0.3` | crítico |
| `budget_lost` | `budget_lost_is != null && budget_lost_is > 0.2 && budget_lost_is <= 0.4` | atenção |
| `budget_lost_alto` | `budget_lost_is != null && budget_lost_is > 0.4` | crítico |
| `sem_conversao` | `conversions === 0 && cost >= max(meta_metrica_mae || 0, 50)` | crítico |
| `ctr_baixo` | **DEFERIDO** — precisa benchmark por objetivo/canal, não disponível no payload hoje. Backlog L3.1+ (ingestão de tabela de benchmarks). | — |

Notas:
- `sem_conversao` usa **piso de R$50** (ou a meta, se maior) para não disparar
  em campanhas com gasto trivial (ex.: R$3 gastos, 0 conversões — ruído, não
  sinal).
- Os pares `_acima_meta`/`_muito_acima_meta` (e equivalentes) existem para
  que a severidade de cada flag seja **atômica** (uma flag = uma severidade),
  em vez de um mapa externo flag→severidade que precisa ser mantido em
  sincronia. O Notion `multi_select` absorve as novas opções on-the-fly.

## 4. Decisão — Agregação de severidade (determinística, em código)

```js
const RANK = { info: 0, atencao: 1, critico: 2 };
let severidade = flags.reduce(
  (acc, f) => (RANK[SEVERITY_BY_FLAG[f]] > RANK[acc] ? SEVERITY_BY_FLAG[f] : acc),
  'info'
);
if (qualidade.volume_suficiente === false) severidade = 'info'; // guarda ADR-21 Tema 10
```

- A severidade final é o **pior caso entre as flags ativas** (crítico > atenção
  > info).
- **Guarda de volume:** se `volume_suficiente=false`, a severidade final é
  **sempre `info`** — mesmo que `sem_conversao` (crítico) esteja ativa. Isso
  não é um bug: é a aplicação literal do Tema 10 (ADR-21) — "nenhum pilar
  emite leitura com volume insuficiente; responde VOLUME INSUFICIENTE em vez
  de pontuar". As flags continuam registradas (auditabilidade), só a
  severidade exibida ao gestor é neutralizada, e o `insight` deve deixar
  isso explícito.
- **A severidade NÃO é decidida pelo LLM.** O LLM recebe `severidade_base`
  (já calculada) e deve **reproduzi-la** em seu output — isso é só para
  permitir uma checagem de consistência (log de divergência, não ação); o
  valor gravado na property Notion `severidade` sempre vem do código
  determinístico, nunca do LLM.

## 5. Decisão — Recomendações

- **Formato:** `{ acao, racional, esforco: baixo|medio|alto, impacto_esperado: baixo|medio|alto }`.
- **Quantidade máxima: 3**, ordenadas por impacto esperado (maior primeiro).
- **Guarda de volume insuficiente:** se ativa, no máximo 1 recomendação, e
  só do tipo "continuar observando" / "ampliar janela de coleta" — nunca
  orçamento, pausa, ou mudança estrutural.
- **Ligação com o loop de Demandas (ADR-22): NÃO nesta iteração.** As
  recomendações ficam como conteúdo narrativo na page de `PHI - ANÁLISES`
  (body). Criar automaticamente uma Tarefa a partir de uma recomendação é
  uma integração de **L3.5** (fora do escopo do L3.0, que já exclui
  explicitamente o loop de Demandas no design original) — decidir isso
  exigiria também resolver o gate cognitivo de "2 leituras consecutivas"
  do ADR-22, que opera em cima de séries temporais de score, não de uma
  análise pontual como esta.

## 6. Decisão — Guardas cognitivas (codificação exata)

Referência: ADR-21 (Temas 10/19/05).

| Guarda | Codificação |
|---|---|
| **Tema 10 — janela estatística / volume insuficiente** | Flag `volume_insuficiente` (regra determinística) → severidade final forçada a `info` (§4) + recomendações limitadas a 1, não-agressiva (§5) + `insight` obrigatoriamente inicia com "VOLUME INSUFICIENTE" (regra de prompt, §2.1 item 7). |
| **Tema 19 — ruído vs problema real** | Sem 2ª leitura de `t28` disponível no payload atual (decisão §8 abaixo resolve isso via proxy TSS, não via nova query). Flag `tendencia_instavel` (TSS < 40) não eleva severidade, mas **rebaixa a confiança do LLM para `hipotese`** — o diagnóstico é sinalizado como preliminar em vez de definitivo. |
| **Tema 05 — não otimizar 1 KPI isolado** | Codificado estruturalmente: o LLM sempre recebe o vetor completo (métrica-mãe + impression_share + budget_lost_is + 6 componentes do score) e a instrução explícita (§2.1 item 5) de ler o vetor antes de concluir. As flags também são plural por natureza — uma única flag ativa não é, por si, instrução para recomendar. |

## 7. Decisão — Modelo Claude + credencial

- **Modelo:** Claude Sonnet (linha atual de modelos Sonnet — confirmar o
  identificador exato disponível no momento de configurar o node Anthropic
  do n8n; modelos mudam e o node deve apontar para o Sonnet vigente, não
  para um ID fixado aqui). Racional: a parte que exige julgamento pesado
  (severidade, flags) já é determinística; o LLM só narra, prioriza e
  recomenda dentro de um schema fechado — não precisa do modelo mais caro
  (Opus) da família. Se o smoke mostrar qualidade insuficiente na
  priorização/narrativa, escalar para Opus é a única mudança necessária
  (troca de 1 parâmetro no node, sem redesenho).
- **Não usar Gemini aqui** — Gemini é reservado para L3.1 (ad/adset, alto
  volume/baixo custo), conforme decisão já travada no design original.
- **Credencial Anthropic no n8n:** continua bloqueante para este passo
  (pendência já registrada em ESTADO-DO-PROJETO §5). Configurar antes do
  Codex implementar o node.
- **Structured Output:** usar tool-use forçado (schema no §10) — não parsing
  de texto livre. Evita o custo de um Output Parser separado e garante que
  o JSON sempre bate com o schema.

## 8. Decisão — Tendência (delta D-7 vs D-7 anterior)

**Decisão: NÃO adicionar nova leitura de `t28` agora.** Em vez disso, reusar
o componente **TSS** (Trend Stability Score) que **já vem no payload** via
`score.tss` — ele já mede exatamente "CPA médio 3d vs CPA médio 7d" (PHI SOP/
Glossário §2). TSS baixo é, por construção, um proxy direto de instabilidade/
ruído recente, sem exigir nenhuma query nova no Orquestrador.

- Isso resolve a pergunta #8 do bootstrap sem aumentar o escopo do plumbing.
- Um verdadeiro delta "D-7 atual vs D-7 anterior" (2ª leitura materializada
  de `t28_campaign`) fica em backlog para quando o gate de "2 leituras
  consecutivas" do ADR-22 (loop de Demandas) for implementado — esse é o
  lugar arquitetural certo para série temporal explícita, não a análise
  pontual do L3.0.

## 9. Bugs a corrigir junto (achados na leitura do JSON real do sub-WF)

1. **`numeric(null) → 0` (já registrado em ESTADO-DO-PROJETO):** o helper em
   `Build Deterministic Flags` faz `Number(null) === 0` (finito) → gera
   `impression_share_baixo` espúrio quando `impression_share` é `null`.
   Fix: `const numeric = (v) => (v == null ? null : (Number.isFinite(Number(v)) ? Number(v) : null));`
2. **Colisão de nome `analise.leitura` (NOVO — achado nesta revisão):** o
   node `Build Deterministic Flags` grava o texto do insight placeholder no
   campo `analise.leitura`, e `Build Notion Page` usa esse mesmo campo como
   `blocks[0].text`. Só que `leitura` também é o **nome da property Notion**
   que recebe `score.phi_classification` (CRITICAL/WARNING/GOOD/EXCELLENT).
   São duas coisas semanticamente diferentes usando o mesmo nome de campo no
   payload interno — risco real de um dev futuro (ou o próprio LLM node)
   confundir os dois. **Fix:** renomear `analise.leitura` → `analise.insight`
   em `Build Deterministic Flags` e no consumo em `Build Notion Page`
   (`blocks[0].text = analise.insight`). Isso também alinha com o nome de
   campo já usado no §10 abaixo e no bootstrap original (§5).

### Mapeamento → `PHI - ANÁLISES` (o que muda vs. o placeholder)

| Campo Notion | Origem (framework) | Origem (placeholder antigo) |
|---|---|---|
| `flags_ativas` | `analise.flags` (determinístico, **inalterado**) | idem |
| `severidade` | `analise.severidade` (determinístico, **inalterado**) | idem |
| Body — Insight | `llm.insight` (**novo**, substitui `analise.insight`/antigo `analise.leitura`) | `analise.leitura` (templated) |
| Body — Recomendações | `llm.recomendacoes` formatadas (**novo**, deixa de estar vazio) | vazio (placeholder) |
| Body — Evidências | `llm.evidencias` (**novo**, LLM-gerado) ou mantém o bloco determinístico atual como fallback se o LLM falhar | bloco determinístico fixo |
| `modelo_llm` | `"claude-<versao>"` (ex.: `claude-sonnet-4-x`, string exata do node) | `placeholder_deterministico_l3_0` |

`flags_priorizadas` e `confianca` do LLM **não** têm property própria no DB —
ficam só no body (ex.: linha "Confiança: hipótese" antes do insight, e a
ordem de `flags_priorizadas` guia a ordem de leitura do texto). Se o gestor
achar útil ter `confianca` como coluna filtrável, é upgrade trivial de schema
depois (não bloqueia esta entrega).
