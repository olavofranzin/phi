# A.7b - DDL completo + reescrita da VIEW `phi_score_current`

## Status
Aguardando aceite.

## Contexto
Este arquivo substitui o blocker report inicial da A.7b.

O bloqueio anterior existiu corretamente: a `VIEW` `phi_score_current` usa `MAX(calculated_date) + GROUP BY + INNER JOIN`, o que parecia ambiguo para propagar `source_execution_id`.

O Ramo A foi liberado depois que o probe canonico retornou 0 linhas para duplicidade por:

- `client_id`
- `campaign_id`
- `calculated_date`

Com isso, a `VIEW` ficou deterministica no estado real de hoje e o DDL pode seguir sem alterar a semantica atual.

### 2. Reescrita da VIEW
Primeira tentativa falhou por sintaxe ao usar identificador com hifen no nome do projeto dentro do `CREATE VIEW`.

Tentativa descartada:

- execution id: `4964`
- status: `error`
- erro: `Syntax error: Expected keyword AS but got "e7c58d4" at [1:33]`

O ajuste foi apenas sintatico: usei notacao de dataset `phi_prod.*` no `CREATE OR REPLACE VIEW`, preservando a logica, a ordem dos campos existentes, os filtros e o `INNER JOIN` originais.

Statement final aplicado:

```sql
CREATE OR REPLACE VIEW phi_prod.phi_score_current AS
SELECT
  h.execution_id,
  h.source_execution_id,
  h.client_id,
  h.campaign_id,
  h.calculated_date,
  h.business_model,
  h.model_id,
  h.model_version,
  h.phi_value,
  h.mas,
  h.tss,
  h.fis,
  h.es,
  h.rs,
  h.os,
  h.threshold_used,
  h.miv,
  h.miv_normalizado,
  h.priority_score,
  h.phi_classification,
  h.calculation_status,
  h.calculation_last_step,
  h.mas_janela,
  h.rs_data_insufficient,
  h.os_data_unavailable,
  h.reprocessed,
  h.reprocessed_at,
  h.reprocessed_by,
  h.snapshot_timestamp
FROM `phi_prod.phi_score_history` h
INNER JOIN (
  SELECT client_id, campaign_id, MAX(calculated_date) AS max_date
  FROM `phi_prod.phi_score_history`
  WHERE calculation_status = 'SUCCESS'
  GROUP BY client_id, campaign_id
) latest
  ON h.client_id        = latest.client_id
  AND h.campaign_id     = latest.campaign_id
  AND h.calculated_date = latest.max_date
WHERE h.calculation_status = 'SUCCESS';
```

## Schema reference atualizado
Going-forward, a semantica aprovada fica:

- `execution_id`: `EXEC-PHI-*` quando a A.6 entrar e o PHI parar de usar `FALLBACK-*`
- `source_execution_id`: `EXEC-DE-*` do snapshot do `Daily Entry` consumido pelo PHI

Estado atual apos A.7b:

- novas execucoes ainda podem gravar `execution_id = FALLBACK-*` porque a A.6 ainda nao foi aplicada
- `source_execution_id` ja existe no schema, mas fica `NULL` ate a A.6 comecar a popular
- historico retroativo permanece `NULL` por decisao explicita de nao fazer backfill

## Nota tecnica obrigatoria: determinismo hoje e tie-breaker futuro
Determinismo hoje:

- o probe rerodado na entrega confirmou `0` linhas duplicadas `SUCCESS` para `(client_id, campaign_id, calculated_date)`
- portanto a `VIEW phi_score_current` continua deterministica hoje mesmo usando apenas `MAX(calculated_date)`

Risco futuro:

- se reprocessamento ou incidente operacional criar duas linhas `SUCCESS` para a mesma chave `(client_id, campaign_id, calculated_date)`, a `VIEW` volta a ficar semanticamente ambigua

Regra candidata de tie-breaker para uma entrega futura:

1. `QUALIFY ROW_NUMBER() OVER (PARTITION BY client_id, campaign_id ORDER BY calculated_date DESC, snapshot_timestamp DESC, execution_id DESC) = 1`
2. manter `calculation_status = 'SUCCESS'`
3. documentar explicitamente a precedencia de `snapshot_timestamp` sobre `execution_id`

Nao apliquei esse tie-breaker agora porque seria mudanca de logica, e isso esta fora do escopo aprovado da A.7b.
